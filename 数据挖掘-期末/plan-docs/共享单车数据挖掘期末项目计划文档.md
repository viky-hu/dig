# 共享单车数据挖掘期末项目计划文档

## 一、项目基本信息

### 1. 项目名称

上海摩拜共享单车出行规律分析与热点区域识别

### 2. 项目目标

本项目基于上海摩拜共享单车骑行数据，完成从数据清洗、探索性分析、空间热点识别、需求预测到前端展示的完整数据挖掘流程。项目重点不放在复杂系统开发，而是用较少学习成本完成一个结构完整、结果清晰、便于展示的期末数据挖掘项目。

### 3. 项目定位

本项目不是完整的共享单车调度平台，而是一个数据分析与可视化展示项目。主要目标是回答以下问题：

1. 上海摩拜共享单车在一天中哪些时间段使用最频繁？
2. 工作日和周末的用车规律是否存在差异？
3. 哪些空间区域是共享单车使用热点？
4. 是否可以根据时间特征预测小时级订单量？
5. 如何用一个简洁的前端页面展示数据挖掘结果？

---

## 二、数据部分

### 1. 数据来源

使用已经准备好的上海摩拜共享单车样本数据文件：

`mobike_shanghai_sample_updated.csv`

该数据为共享单车骑行记录数据，适合用于交通出行场景下的数据挖掘分析。

### 2. 数据字段

主要字段包括：

| 字段名 | 含义 | 用途 |
|---|---|---|
| orderid | 订单编号 | 唯一标识一条骑行记录 |
| bikeid | 单车编号 | 统计车辆使用情况 |
| userid | 用户编号 | 统计用户数量和行为 |
| start_time | 骑行开始时间 | 时间规律分析、订单量预测 |
| start_location_x | 起点经度 | 空间热点分析 |
| start_location_y | 起点纬度 | 空间热点分析 |
| end_time | 骑行结束时间 | 计算骑行时长 |
| end_location_x | 终点经度 | OD 流向分析，选做 |
| end_location_y | 终点纬度 | OD 流向分析，选做 |
| track | 骑行轨迹 | 本项目可不深入处理，作为扩展内容 |

### 3. 数据使用策略

为了降低学习成本，本项目不再额外获取 API 数据、不爬取新数据、不更换数据集。当前数据字段已经能够支持时间分析、空间聚类和简单预测建模。

## 四、数据清洗模块

### 1. 负责人建议

适合由 1 名成员负责。

### 2. 主要任务

数据清洗是后续分析和建模的基础，主要完成以下工作：

1. 读取 CSV 数据文件
2. 查看数据规模、字段类型和缺失值情况
3. 将 `start_time` 和 `end_time` 转换为时间格式
4. 计算骑行时长，单位建议为分钟
5. 提取时间特征：
   - 日期
   - 小时
   - 星期几
   - 是否周末
6. 检查并处理异常数据：
   - 骑行时长小于 1 分钟的数据
   - 骑行时长过长的数据，例如超过 180 分钟
   - 起终点坐标明显异常的数据
7. 保存清洗后的数据文件

### 3. 输入文件

`mobike_shanghai_sample_updated.csv`

### 4. 输出成果

| 输出内容 | 文件或形式 |
|---|---|
| 清洗后的数据 | `cleaned_mobike.csv` |
| 数据字段说明 | 表格 |
| 数据清洗说明 | 报告文字 |
| 缺失值和异常值统计 | 表格或截图 |

### 5. 报告中可写内容

报告中需要说明：

1. 原始数据包含哪些字段
2. 时间字段如何处理
3. 骑行时长如何计算
4. 异常值如何判断和过滤
5. 清洗后数据可以支持哪些分析任务

## 六、空间热点分析模块

### 1. 负责人建议

适合由 1 名成员负责。

### 2. 主要任务

该模块用于识别共享单车高频使用区域，体现数据挖掘中的聚类分析方法。

需要完成以下工作：

1. 选取起点坐标：
   - `start_location_x`
   - `start_location_y`
2. 使用 KMeans 聚类算法识别热点区域
3. 设置聚类数量 K，建议使用 6 或 8
4. 为每条订单记录添加起点聚类标签
5. 统计每个热点区域的订单数量
6. 绘制空间散点图
7. 标出聚类中心点

### 3. 推荐模型

使用 KMeans 聚类模型即可。

选择原因：

1. 学习成本低
2. scikit-learn 中调用简单
3. 聚类结果容易可视化
4. 符合数据挖掘课程要求

### 4. 输出成果

| 输出内容 | 建议形式 |
|---|---|
| 聚类后的数据 | 添加 cluster 字段的数据表 |
| 热点区域统计 | 每个 cluster 的订单数量表 |
| 空间热点分布 | 经纬度散点图 |
| 聚类中心 | 表格或图中标记 |

### 5. 报告中可写内容

报告中需要说明：

1. 为什么使用 KMeans 聚类
2. 使用了哪些字段作为聚类特征
3. 聚类数量如何设置
4. 哪些区域订单量较高
5. 热点区域对共享单车投放和运营有什么参考意义

## 八、预测建模模块

### 1. 负责人建议

适合由 1 名成员负责。

### 2. 建模目标

预测每小时共享单车订单量。

该目标比预测用户目的地、车辆调度或骑行轨迹简单，更适合时间有限的期末项目。

### 3. 数据构造方式

将原始订单数据按小时聚合，得到小时级订单量数据。

示例结构：

| datetime_hour | hour | weekday | is_weekend | order_count |
|---|---|---|---|---|
| 2016-08-01 08:00 | 8 | 0 | 0 | 订单数 |

其中：

- `order_count` 为预测目标
- `hour`、`weekday`、`is_weekend` 为输入特征

如果时间允许，可以增加：

- 前一小时订单量 `last_hour_count`
- 前一天同小时订单量 `last_day_same_hour_count`

### 4. 推荐模型

至少完成一个模型，推荐完成两个模型做对比：

1. LinearRegression
2. RandomForestRegressor

推荐最终主模型使用 RandomForestRegressor。

原因：

1. 使用简单
2. 对非线性规律有较好表现
3. 不需要复杂参数调优
4. 结果容易解释和展示

### 5. 评价指标

使用以下指标即可：

| 指标 | 含义 |
|---|---|
| MAE | 平均绝对误差 |
| RMSE | 均方根误差 |
| R² | 模型拟合程度 |

### 6. 输出成果

| 输出内容 | 建议形式 |
|---|---|
| 建模数据集 | 小时级订单量表 |
| 模型评价结果 | 指标表 |
| 真实值与预测值对比 | 折线图 |
| 模型结论 | 报告文字 |

## 十四、项目风险与简化方案

### 1. 如果时间不足

优先完成以下内容：

1. 数据清洗
2. 时间规律分析
3. KMeans 热点聚类
4. Streamlit 简单展示
5. 报告

可以暂时不做：

1. OD 流向分析
2. 多模型对比
3. 复杂参数调优
4. 复杂地图展示

### 2. 如果模型效果一般

可以在报告中说明：

1. 当前模型只使用了简单时间特征
2. 未引入天气、节假日、地铁站点等外部因素
3. 因此预测结果存在一定误差
4. 后续可以加入更多外部变量提升模型效果

### 3. 如果前端开发时间不够

可以将 Streamlit 页面压缩为 4 个部分：

1. 数据概览
2. 时间分析
3. 热点聚类
4. 模型预测

只要图表能显示、结论清楚，就可以满足期末项目展示需要。
